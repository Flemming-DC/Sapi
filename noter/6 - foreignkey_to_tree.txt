if a parent has a non-unique fk to a child, then multiple parents could share the same child, which 
breaks the hieracical structure for rows. This breaks inserts into tree.
Note that the tables can still be given a hieracial structure, just not the rows. Ergo auto-joins are ok.
Thus selects are still ok. 
I don't see a problem with updates.
If a parent refers to a child with a not null fk, then deleting the child forces a delete on the parent,
which can be surprising a therefore cause accidental deletion of data. Thus such a tree is non-deletable.
A parent could have an unchecked reference to a child (thats a sapi-only thing) and this is ok with deletion.

rules:
    1) Any tree must either have a hierachic datamodel or acquire one via circle-breaking (implemented by 
       join-deletion or normal join rules). This hierachy is between tables, not rows. 
    2) An insertable tree must also be hierachic at the row level. This condition is satsified iff no
       parent has a non-unique reference to a child.
    3) A deletable tree cannot contain any parent with a checked not null reference to a child.
    4) Every tree is updatable and selectable.

insertable: no parent has a non-unique reference to a child.
deletable: no parent with a checked not null reference to a child.

let a full tree be a selectable, insertable, updatable and deletable tree. 
full tree: no parent has a (non-unique or a checked not null) reference to a child.
